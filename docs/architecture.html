  1. ① 电话接入 — 路由分配

    Twilio 触发 webhook（`/twilio/voice`），FastAPI 把这个 WebSocket 升级为 `TwilioCall` 实例；浏览器那边则直接 `BrowserSession.run()` accept WS。

    app/main.py · app/services/{twilio_handler,browser_session}.py

  2. ② STT 开始转写 — 边听边记

    音频字节流送进 `make_transcriber()` 选出的 STT（百炼 Paraformer 优先）。STT 在 WebSocket 里推 `TranscriptEvent`（interim / final），后台 task drain 到 `messages` 队列。

    app/services/stt.py:527 · make_transcriber()

  3. ③ 用户说完一段 — 推给 LLM

    VAD 检测到 1.44s 静音（`_on_pcm` 的服务端静音帧算法）→ STT 标 `speech_final` → `push_user_utterance(text)` 把这段写进 agent 的 OpenAI `messages` 列表。

    app/services/browser_session.py:276 · VisitorLLMAgent.push_user_utterance:91

  4. ④ LLM 决定下一步 — tool calling

    `step()` 调 `chat.completions.create(tools=TOOLS_SCHEMA)`，LLM 可能调 `update_visitor_info` 填一两个槽，或者直接调 `send_to_guard_and_end_call` 挂电话。每个 tool 在 `app/tools/__init__.py` 里有独立 handler。

    app/agent.py:99 step() · app/tools/__init__.py:145 TOOLS_REGISTRY

  5. ⑤ TTS 把回复念出来

    LLM 吐的 `text_to_speak` 送进 `synthesize_to_*` —— `pick_provider()` 选最佳 TTS（百炼 CosyVoice 流式优先），转成 μ-law 8kHz（Twilio）或 PCM 16kHz（浏览器）推回客户端。

    app/services/tts.py:192 pick_provider · :220 synthesize_to_mulaw8k

  6. ⑥ 结束 — 通知门卫 + 挂电话

    当 LLM 调 `send_to_guard_and_end_call`（或者后置保险 `_looks_like_goodbye` + `is_complete()` 命中）→ `to_wechat_card()` 渲染 Markdown 卡片发到门卫群 → Twilio/Browser 收到 `is_final=True` → 挂电话。`VisitorInfo` 落 SQLite 留档。

    app/schemas.py:92 to_wechat_card · app/services/wechat.py

§ 6 · 深入

Function Calling · Schema 设计 & 一次完整 call

把 app/tools/__init__.py 的 schema 怎么写、registry 怎么 dispatch、LLM 返回后每步发生什么，拆到一行行讲清楚。

6.1 一个 Tool 的 schema 长啥样

OpenAI Function Calling 的 schema 就是 JSON Schema 子集，包成两层 {type:"function", function:{...}}。5 个字段逐个说：

type固定的 wrapper
"type": "function"   // OpenAI 协议规定，永远是 "function"
function.name调用的方法名
"name": "update_visitor_info"   // 模型用这个名调你，必须匹配 handler 注册名
function.description最关键 — LLM 决定何时调
"description": "Incremental update of visitor info slots. Call this as soon as you hear ANY field. Do not wait until the end. You may call multiple times per turn."   // 大模型靠这段判断什么时候调
function.parametersJSON Schema 对象
"parameters": { "type": "object", "properties": {...}, "required": [...] }
handler业务逻辑（不在 schema 里）
handler: _handle_update_visitor_info   // 收到 LLM 的 args 后干啥
💡
注意：前 4 个字段是给 LLM 看的（决定它会不会调、调什么参数），最后那个 handler 是给 你的 Python 看的（决定收到调用后干啥）。这是 OpenAI 的设计：声明和实现是分离的。

6.2 description 怎么写才不翻车

LLM 完全靠 description 决定何时调、调哪个。对比一下：

❌ 烂的 description

"Update visitor info."

→ LLM 不知道什么时候该调、什么时候不该调，可能每次都调、或者永远不调。

✅ 好的 description（项目真实写法）

"Incremental update of visitor info slots. Call this as soon as you hear ANY field. Do not wait until the end. You may call multiple times per turn."

→ 告诉 LLM：触发时机（听到任一字段就调）、调用频次（一轮可多次）、反模式（不要等到最后）。

⚠️
经验法则：description 里出现 "CRITICAL" / "EXACTLY ONCE" / "the ONLY way" 这种大写强制词时，模型严格遵守率显著更高——见本项目 send_to_guard_and_end_call 的写法。

6.4 step() 收到 tool_call 后每步发生什么

    
从 LLM 返回到 AgentReply 生成，6 个步骤串成一条线：


    
    
      
  1. 
        LLM 返回 msg.tool_calls 列表
        
    每个 tc 有 id、function.name、function.arguments（JSON 字符串，要 parse）。
    
      
    2. 
      
  2. 
        把 assistant 消息写进 self.messages
        
    带 tool_calls 字段。这是 OpenAI 协议规定的——assistant 说了什么，history 必须原样记。
    
      
    3. 
      
  3. 
        遍历每个 tc，从 TOOLS_REGISTRY 查 handler
        
    tool = TOOLS_REGISTRY.get(tc.function.name)。查不到就 warn + 返回错误 result（不让 LLM 卡死）。
    
      
    4. 
      
  4. 
        调 await tool.handler(args, ctx)
        
    handler 拿到 args: dict 和 ctx: ToolContext（含 visitor、is_final、summary），返回 result: dict。
    
      
    5. 
      
  5. 
        把 result 写进 self.messages，role: "tool"
        
    必须带 tool_call_id 关联到 step 2 的 assistant tool_call。下一轮 LLM 就能看到自己调的工具返回了什么。
    
      
    6. 
      
  6. 
        读 ctx.is_final 和 ctx.summary，返回 AgentReply
        
    handler 改 ctx，agent 读 ctx——这个解耦是 tools 包重构的核心收益。

async def _handle_lookup_plate_history(
    args: dict[str, Any], ctx: ToolContext,
) -> dict[str, Any]:
    plate = args.get("plate", "").strip()
    if not plate:
        return {"ok": False, "error": "plate required"}

    # 1) 走项目已有的 SQL 查历史
    from app import sql as sql_module
    history = await sql_module.plate_history(plate, days=365)

    # 2) 命中 → 标回访 + 把历史装进 ctx.visitor，
    #    让 is_complete() 自动走"只缺车牌"捷径
    if history:
        ctx.visitor.is_returning = True
        ctx.visitor.call_history = [
            VisitorInfo.model_validate(h) for h in history
        ]

    # 3) 返回给 LLM（最多 5 条避免 context 爆炸）
    return {
        "ok": True,
        "is_returning": bool(history),
        "count": len(history),
        "recent": [
            f"{v['started_at']:%Y-%m-%d} · {v['reason']} · {v['company']}"
            for v in history[:5]
        ],
    }

6.6 常见坑 · 提前避雷


    

      🚫
      
坑 1：description 太短。LLM 看到 "Update visitor info" 完全不知道何时调——结果是它要么永不调、要么每次都调。加触发时机、调用频次、反模式三件套。

    


    

      🚫
      
坑 2：忘了写 required 字段。LLM 传 {} 空对象也能"调通"——你的 handler 必须自己处理 args.get("xxx") is None 的情况。本项目 update_visitor_info 的所有字段都是 optional，handler 用 if v: setattr(...) 跳过 None；如果是 required，handler 必须返回 {"ok": False, "error": "missing plate"}。

    


    

      🚫
      
坑 3：handler 抛异常没接住。如果 handler 内 await something() 抛错，agent.step() 整个 step 都会挂。解法：handler 内 try/except + 返回 {"ok": False, "error": str(e)}，让 LLM 看到错误后重试或换策略。

    


    

      🚫
      
坑 4：忘了写 role: "tool" 消息。每条 assistant.tool_calls 都必须有对应的 tool 消息回复，否则下一轮 LLM 报协议错。重构后的 step() 末尾已经统一 append 了，但写新 tool 时别忘了 ctx 是要返回 result 的。

    


    

      🚫
      
坑 5：max_completion_tokens 太小。本项目设的 200——够 LLM 说一句中文 + 调 1-2 个 tool。如果有 tool 的 description 巨长或 properties 巨多，模型可能没空间生成 tool_calls JSON。调到 300-500 看看。

"""Embedding client. BGE-small-zh 是中文 SOTA，~93MB，本地推理 50ms/chunk。"""
from sentence_transformers import SentenceTransformer
from functools import lru_cache

@lru_cache
def _model():
    return SentenceTransformer("BAAI/bge-small-zh-v1.5")

async def embed(texts: list[str]) -> list[list[float]]:
    return _model().encode(texts, normalize_embeddings=True).tolist()

"""把 docs/ 下的 SOP PDF/Markdown 切成 chunks → embed → 写 Chroma。
冷启动时跑一次，之后增量更新即可。"""
import chromadb
from app.rag.embeddings import embed

def build_index(docs_dir: str = "docs/sop"):
    client = chromadb.PersistentClient(path=".chroma/")
    col = client.get_or_create_collection("sop", metadata={"hnsw:space": "cosine"})

    chunks = []
    for path in Path(docs_dir).rglob("*.md"):
        chunks += split_md(path, chunk_tokens=500, overlap=100)
    # split_md 自己按段落/标题切，保持语义完整

    embs = await embed([c["text"] for c in chunks])
    col.add(
        ids=[c["id"] for c in chunks],
        embeddings=embs,
        documents=[c["text"] for c in chunks],
        metadatas=[c["meta"] for c in chunks],   # 包含 source_file / section
    )

      

    


    

      
3app/rag/retrieve.py · 在线检索›

      


"""Top-k cosine 检索，~30ms 含 embed。"""
import chromadb
from app.rag.embeddings import embed

_col = chromadb.PersistentClient(path=".chroma/").get_collection("sop")

async def query_sop(question: str, k: int = 5) -> list[dict]:
    q_emb = (await embed([question]))[0]
    res = _col.query(query_embeddings=[q_emb], n_results=k)
    return [
        {"text": doc, "source": meta["source_file"], "score": 1 - dist}
        for doc, meta, dist in zip(
            res["documents"][0], res["metadatas"][0], res["distances"][0]
        )
    ]

5边界控制 · app/rag/guard.py（常被忽略的雷）›

      

        
RAG 最常见的失败模式：检索到了不相关 chunk，LLM 把它当真。所以必须做：


async def _handle_query_sop(args, ctx):
    from app.rag.retrieve import query_sop
    hits = await query_sop(args["question"], k=5)

    # ① 相似度阈值 — 太低就拒答
    if not hits or hits[0]["score"] < 0.5:
        return {"ok": True, "matches": [], "note": "no relevant SOP found"}

    # ② 截断 — 防 prompt 爆炸
    top3 = hits[:3]

    # ③ 来源标注 — 让 LLM 知道"这是从哪条规则来的"
    return {
        "ok": True,
        "matches": [
            {"source": h["source"], "text": h["text"][:300], "score": h["score"]}
            for h in top3
        ],
    }

        
同时在 system prompt 里加："query_sop 返回的内容仅作参考，最终判断需结合上下文；找不到就直说不知道。"

Q4CosyVoice 流式 TTFB ~200ms 是怎么做到的？其他 provider 为什么不行？›

      

        
考点：对 streaming vs one-shot 架构差异的理解。


参考答法：
"CosyVoice WebSocket 是 server-side streaming——run-task 之后服务端
立即开始推 audio chunk，不用等服务端把整段音频生成完。这是协议级
streaming，和 OpenAI 那种 'response_format=mp3 然后等整个文件' 是
两回事。
qwen3-tts-flash 走 REST one-shot，必须等生成完才能下载 WAV，
